# 产品宣传白皮书设计说明

> 日期：2026-03-31
> 角色：Leader
> 适用对象：对外展示、客户交流、验收汇报、项目归档附件

---

## 1. 目标

为“三方比价支出依据扫描工具”产出一套可重复迭代的白皮书交付方案，满足以下目标：

- 以产品宣传白皮书风格对外展示项目价值与成熟度
- 基于 Markdown 编写，便于版本管理和持续迭代
- 通过 Python 脚本稳定导出 Word（`.docx`）交付件
- 使用真实界面截图作为能力证据，并统一存放、统一命名、统一嵌入

本次交付重点不是技术设计归档，而是让外部读者快速理解“产品解决什么问题、具备什么能力、如何运行、为何可信、交付到了什么程度”。

## 2. 目标读者

优先读者：

- 客户方业务负责人
- 项目验收参与方
- 采购、审计、财务等业务代表
- 管理层和非研发背景决策者

次级读者：

- 售前、实施、交付人员
- 需要快速了解系统能力边界的合作方

因此文档表达要满足两点：

- 先讲业务价值和流程，再讲架构与技术
- 技术内容用于建立可信度，而不是展开实现细节

## 3. 风格定位

文档整体采用“产品宣传白皮书”风格，要求：

- 语言专业、克制、可信，不写夸张宣传语
- 结构清晰，适合打印、邮件发送、作为验收附件
- 图文结合，页面节奏以“概述 + 能力 + 证据”为主
- 技术描述聚焦本地化、安全性、稳定性、可追溯性，不展开代码实现

不采用以下风格：

- 不写成 PRD 重述版
- 不写成研发技术方案评审稿
- 不写成用户操作手册

## 4. 核心叙事

白皮书建议围绕以下主线展开：

1. 企业在三方比价、支出依据整理、采购审计留痕方面存在人工汇总成本高、追溯难、口径不一致的问题
2. 本系统以本地桌面端方式，覆盖文件导入、标准化、归组、符合性审查、比价导出全流程
3. 系统强调高可控、可追溯、可修正、可导出，适合采购、审计、财务日常业务和项目验收场景
4. 产品已具备可演示、可验收、可交付的完整链路，并通过真实界面和导出结果展示成熟度

## 5. 建议章节结构

### 5.1 封面

- 产品名称
- 副标题
- 版本号
- 日期
- 公司或项目标识（如有）

建议副标题：

“面向采购、审计与财务场景的本地化三方比价审计辅助平台”

### 5.2 执行摘要

用 1 页以内快速说明：

- 产品解决的问题
- 目标用户
- 核心价值
- 主要能力

### 5.3 建设背景与业务痛点

聚焦外部视角：

- 多家供应商报价文件格式不统一
- 手工比价和整理底稿效率低
- 过程留痕弱，追溯难
- 对合规性和证据管理要求越来越高

### 5.4 产品定位与应用场景

说明适用角色、典型场景和边界：

- 采购专员
- 审计人员
- 财务人员
- 项目经理

### 5.5 核心功能总览

按模块概述而非按内部技术模块展开：

- 文件导入与解析
- 字段标准化与人工修正
- 商品归组与人工确认
- 需求符合性审查
- 比价结果生成与 Excel 导出

### 5.6 五阶段业务流程

这是正文主章节之一，建议配多张截图：

- 导入文件
- 标准化
- 商品归组
- 符合性审查
- 比价导出

### 5.7 关键产品亮点

建议提炼 5-7 个亮点：

- 完全本地运行
- 可追溯证据链
- 关键步骤可人工修正
- 支持多格式报价文件
- 支持符合性矩阵
- 支持双口径最低价
- 支持审计底稿导出

### 5.8 系统架构与运行方式

只讲外部可理解的架构要点：

- Tauri 桌面壳
- React 前端
- Python Sidecar
- SQLite 项目存储
- 本地 HTTP 通信与会话保护

### 5.9 数据安全与部署优势

强调：

- 本地存储
- 无外网依赖
- 项目数据自包含
- 适合企业内网和敏感数据处理

## 9. 交付物定义

本轮白皮书工作最终应形成以下交付物：

- Markdown 源文件：`docs/whitepaper/software-whitepaper.md`
- 截图资源目录：`docs/whitepaper/assets/screenshots/`
- 图片清单说明：`docs/whitepaper/assets/screenshots/README.md`
- Word 导出脚本：`scripts/export_whitepaper_docx.py`
- Word 输出文件：`dist/whitepaper/software-whitepaper.docx`

## 10. 成功标准

当满足以下条件时，可认定白皮书方案达成：

- 文档风格符合对外宣传展示与验收汇报用途
- 正文可完整说明产品价值、核心流程和交付成果
- 关键章节有真实截图支撑
- Markdown 可稳定转换为 Word
- Word 可直接交付，不需要大规模手工重排

## 11. 风险与对策

### 风险 1：真实截图状态不一致

对策：

- 固定演示项目和窗口尺寸
- 按截图清单逐项拍摄
- 不足部分用设计稿补位，但需在内部标记来源

### 风险 2：Markdown 转 Word 后版式失真

对策：

- 先控制 Markdown 结构简单稳定
- 图片、表格、标题层级使用统一规范
- 脚本只承担稳定转换，不追求复杂排版

### 风险 3：内容偏技术、缺少对外表达

对策：

- 每章先写业务价值，再写能力说明
- 用“应用场景、流程、结果”组织内容
- 把深技术内容压缩为 1-2 节支撑说明
