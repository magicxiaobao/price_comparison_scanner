## 1. 产品基本信息

### 1.1 产品名称

三方比价支出依据扫描工具
Three-Party Price Comparison Scanner

### 1.2 产品定位

一款 **完全本地运行** 的桌面端采购比价审计辅助工具。
用于帮助采购、审计、财务人员快速处理多家供应商报价文件，并支持可选的需求符合性审查，完成：

- 文件导入
- 表格提取
- 字段标准化
- 商品归组候选生成
- 人工确认
- 项目需求符合性审查（可选）
- 基础比价
- Excel 审计底稿导出

### 1.3 MVP 核心目标

在本地环境下，支持用户完成 **3 家及以上供应商报价文件** 的整理与比价，输出 **可追溯、可校正、可导出的 Excel 审计底稿（含比价结果与符合性矩阵）**。

### 1.4 产品原则

1. **本地运行**：所有数据、规则、项目文件均存储在本地
2. **高可控**：关键判断必须支持人工确认
3. **可追溯**：最终结果必须可追溯到原始文件与处理过程
4. **先稳定后智能**：优先保证流程可用，不追求第一版全自动
5. **准企业产品**：支持企业内部标准化推广，但第一版不做多用户协同

# 3. MVP 范围

## 3.1 MVP 包含范围

### 3.1.1 项目管理

- 新建项目
- 自动保存项目
- 最近项目列表
- 重新打开最近项目

### 3.1.2 文件导入

支持导入以下文件类型：

#### 强支持

- Excel（.xlsx）
- Word（.docx）
- 数字版 PDF

#### 实验性支持（MVP 不作为硬性验收目标，推迟到后续版本完善）

- 图片（JPG / PNG / TIFF）
- 扫描版 PDF

> **决策说明**：OCR 表格识别准确率低、开发成本高，MVP 优先保证结构化文件的完整流程。图片/扫描 PDF 作为实验性能力提供，用户需自行校验结果。

### 3.1.3 文件解析

- **Excel（.xlsx）**：读取工作表中的表格数据
- **Word（.docx）**：提取文档中的表格
- **数字版 PDF**：优先尝试结构化提取（基于 PDF 内嵌文本和表格结构）
- **图片 / 扫描版 PDF（实验性）**：OCR 提取表格内容，结果需人工校验

#### 数字版 PDF 解析 Fallback 策略

数字版 PDF 的结构化提取可能因排版复杂、表格嵌套、跨页表格等原因失败。MVP 采用以下分级降级策略：

| 级别 | 策略 | 触发条件 |
|------|------|----------|
| L1：结构化提取 | 直接解析 PDF 内嵌文本和表格结构 | 默认首选 |
| L2：转图片 OCR | 将 PDF 页面渲染为图片后进行 OCR 表格识别 | L1 未识别到表格，或识别结果为空/明显异常 |
| L3：人工介入 | 提示用户「该文件自动解析失败，建议将内容手动复制到 Excel 后重新导入」 | L2 结果置信度过低或完全失败 |

> **说明**：L2 降级时，系统应自动将该文件标记为「OCR 解析」，对应字段置信度按 OCR 标准标记（低于结构化提取）。L3 不阻断项目流程，用户可跳过该文件继续处理其他供应商。

### 3.1.4 供应商确认

- 系统可根据文件名或内容猜测供应商名称
- 用户必须手动确认或修改供应商名称后，才能进入后续步骤

### 3.1.5 表格选择

- 一个文件中识别出多个表格时，支持用户选择参与比价的表格
- 支持忽略无关表格

### 3.1.6 匹配规则管理

支持本机本用户维度的全局规则管理：

- 新增规则
- 编辑规则
- 删除规则
- 启用/停用规则
- 默认模板加载
- 规则导入/导出（JSON）

#### 第一版规则类型

1. **列名映射规则**
   - 例：价格、单价、报价 → 单价
2. **值标准化辅助规则**
   - 例：交货周期、交期 → 交期

#### 匹配方式

- 精确匹配
- 模糊匹配（可选）
- 正则匹配（高级模式）

#### 规则冲突解决策略

当同一原始列名或关键词命中多条规则时，按以下优先级裁决：

1. **项目级覆盖规则** > **全局用户规则** > **内置默认模板规则**
2. 同一层级内，**精确匹配** > **正则匹配** > **模糊匹配**
3. 同一层级、同一匹配方式下，**后创建的规则** 优先（用户最近意图优先）
4. 若仍存在歧义，系统不自动裁决，将该列标记为「命中多条规则，需人工确认」，由用户在标准化预览中手动选择

### 3.1.7 标准化预览与人工修正

- 将多家供应商表格统一映射为标准字段
- 支持表格预览
- 支持手工修正字段值
- 支持项目内覆盖全局规则
- 支持自动计算总价（数量 × 单价）

#### 标准字段定义

| 字段名 | 数据类型 | 是否必填 | 说明 | 默认映射关键词示例 |
|--------|----------|----------|------|-------------------|
| 商品名称 | 文本 | 必填 | 商品或服务的名称 | 品名、名称、产品名称、物资名称、项目名称 |
| 规格型号 | 文本 | 选填 | 商品规格、型号、配置 | 规格、型号、规格型号、配置、参数 |
| 单位 | 文本 | 必填 | 计量单位 | 单位、计量单位 |
| 数量 | 数值 | 必填 | 采购数量 | 数量、采购数量、需求数量 |
| 单价 | 数值 | 必填 | 单件价格 | 单价、报价、含税单价、不含税单价、价格 |
| 总价 | 数值（可自动计算） | 选填 | 数量 × 单价，若原表未提供可自动计算 | 总价、金额、合计、小计 |
| 税率 | 文本/数值 | 选填 | 增值税税率，如 13%、6% | 税率、增值税率、税点 |
| 交期 | 文本 | 选填 | 交货周期或日期 | 交期、交货期、交货周期、供货周期、货期 |
| 备注 | 文本 | 选填 | 其他补充信息 | 备注、说明、附注 |

> **说明**：「默认映射关键词」为系统内置列名映射规则的初始值，用户可通过规则管理进行新增和修改。「必填」表示该字段为空时系统应高亮提示，但不阻断流程。

### 3.1.8 商品归组候选与人工确认

#### 归组策略（C+ 方案）

系统基于商品名称、规格型号、单位进行 **归一化处理** 和 **候选匹配**，采用「宁可少自动，不可乱自动」的保守策略。

##### 第一步：文本归一化

在匹配前对字段进行标准化预处理：

- **商品名称归一化**：全部转小写、去除多余空格和标点、中英文括号统一、内置品牌别名映射（如 lenovo → 联想）、通用噪音词降权
- **规格型号归一化**：拆分为 token 后比较，不做整串比较
- **单位归一化**：仅做文本统一，不做单位换算；单位不一致时直接降为待确认

> **决策说明**：归一化规则（品牌别名映射、噪音词列表）作为内置规则的一部分，MVP 内置默认集合，不开放用户编辑。后续版本可考虑开放。

##### 第二步：多因子打分分层

基于商品名称相似度（权重最高）、规格型号 token 重合度（次高）、单位一致性（强约束）进行打分，结果分为三档：

| 置信度 | 条件 | 系统行为 |
|--------|------|----------|
| **高置信** | 名称归一化后几乎一致 + 型号核心 token 一致 + 单位一致 | 自动归为候选组，仍允许用户确认/拆分 |
| **中置信** | 名称比较像 + 型号部分重合 + 单位一致 | 放入「待确认候选」，用户一键确认或拆分 |
| **低置信** | 仅名称模糊相似 / 型号不一致或缺失 / 单位不一致 | 不建议合并，保持独立项 |

> **核心原则**：错归组比漏归组更危险。MVP 不使用语义模型（embedding / 语义检索），坚持规则化、可解释、保守的匹配方案。

##### 禁止自动归组的场景（硬约束）

以下任一条件成立时，系统 **不得** 将两条记录自动归为同一候选组，必须保持独立或放入待确认：

1. **单位不一致**：归一化后单位文本不同
2. **规格型号核心 token 冲突**：型号中存在明确不同的核心标识（如 E14 vs E15、i5 vs i7）
3. **商品名称相似但品牌不同**：归一化后品牌名明确不同（如联想 vs 惠普）
4. **任一关键字段为低置信且未经用户确认**：低置信数据不得作为自动归组的依据
5. **数量级明显不一致**：同名商品但数量差异超过 10 倍（可能为不同规格包装）

> **约束说明**：以上场景宁可多产生独立项让用户手工合并，也不可错误归组。具体相似度公式、阈值参数、品牌别名初始表、噪音词初始表等算法细节，见独立技术文档《商品归组算法说明》（技术设计阶段产出）。

##### 用户操作

- 确认归组
- 拆分错误归组
- 手工合并归组
- 标记为不可比项

### 3.1.9 项目需求标准与供应商符合性匹配（可选模块）

#### 模块定位

用户先录入项目需求标准（功能点/技术要求/商务要求），系统基于供应商文档内容进行证据匹配，输出「符合性矩阵」。**该模块为可选能力**：不录入需求标准时，跳过此步骤直接进入比价（退化为纯比价流程）。

> **设计原则**：符合性匹配以「商品组 × 供应商」为对象，而不是单行报价行。只有在商品归组完成后，才能围绕同一商品组判断供应商是否满足要求。

#### 项目需求标准

由用户手工录入或通过模板 Excel 导入，结构化管理。每条需求项包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 需求编号 | 自动生成或用户指定 | REQ-001 |
| 需求分类 | 功能要求 / 技术规格 / 商务条款 / 服务要求 / 交付要求 | 技术规格 |
| 需求描述 | 具体要求内容 | 内存不低于 16GB |
| 是否必选 | 必选 / 可选 | 必选 |
| 判断类型 | keyword（关键词匹配）/ numeric（数值比较）/ manual（人工判断） | numeric |
| 目标值/判断条件 | 关键词、数值范围或人工说明 | ≥ 16 |
| 备注 | 补充说明 | |

#### 供应商符合性匹配结果

针对每个「商品组 × 供应商 × 需求项」，输出匹配结果：

| 状态 | 含义 | 是否参与有效最低价 |
|------|------|-------------------|
| **符合** | 供应商明确满足该需求 | 是 |
| **部分符合** | 供应商部分满足或表述模糊 | 默认否，用户确认「可接受」后参与 |
| **不符合** | 供应商明确不满足 | 否 |
| **无法判断** | 供应商文档中找不到相关信息 | 默认否，用户确认后可调整 |

每条匹配结果必须附带证据：

- 证据原文（从供应商文档中提取的相关文本）
- 证据来源位置（文件名 + 来源定位）
- 匹配方式（keyword / numeric / manual）
- 置信度
- 是否经人工确认

#### MVP 匹配方式

| 类型 | 说明 | 系统行为 |
|------|------|----------|
| keyword | 在供应商标准化数据的规格/备注/名称中搜索关键词 | 命中 → 符合；未命中 → 无法判断 |
| numeric | 从供应商数据中提取数值，与目标值做比较（≥/≤/=/范围） | 满足 → 符合；不满足 → 不符合；无法提取 → 无法判断 |
| manual | 系统不自动匹配，直接标记为「待人工确认」 | 用户手工判定 |

> **核心原则**：与商品归组 C+ 策略一致 —「宁可少自动，不可乱自动」。高确定性结果自动给建议，中低确定性标记待人工确认，不做黑盒裁决。

#### 用户操作

- 录入/导入项目需求标准
- 查看符合性矩阵（横轴供应商，纵轴需求项）
- 确认/修改每条匹配结果
- 对「部分符合」项标记是否「可接受」
- 查看证据详情

### 3.1.10 基础比价

按商品归组输出比价结果，至少包括：

- 各供应商报价
- **全量最低价**（所有供应商参与）
- **有效最低价**（仅「符合」或「部分符合且用户确认可接受」的供应商参与；无需求标准时等同于全量最低价）
- 平均价
- 最高价
- 差额
- 缺项提示
- 异常提示
- 符合性状态摘要（若有需求标准）

> **比价规则变更**：有效最低价默认只在满足需求的供应商范围内计算。「部分符合」默认不参与有效最低价，用户确认「可接受」后才纳入。「不符合」和「无法判断（未确认）」的供应商不参与有效最低价。

### 3.1.11 Excel 导出

导出 Excel 审计底稿，至少包含以下 Sheet（若项目无需求标准，Sheet 4 不生成）：

#### Sheet 1：比价结果表

- 商品组
- 各供应商报价
- 全量最低价高亮
- 有效最低价高亮（若有需求标准）
- 各供应商符合性摘要（若有需求标准：符合/部分符合/不符合/无法判断）
- 平均价
- 差额
- 异常标记

#### Sheet 2：标准化明细表

- 供应商
- 原始列名
- 标准列名
- 标准化后的数据
- 是否人工修改

#### Sheet 3：追溯信息表

- 来源文件名
- 供应商
- 来源定位（Excel：sheet 名 + 单元格坐标；Word：表格索引 + 行列位置；PDF：页码 + 表格索引 + 行列位置）
- 原始列名
- 原始值
- 标准字段名
- 标准化后值
- 命中规则（规则 ID + 规则名称 + 命中时规则内容摘要）
- 匹配方式（精确 / 模糊 / 正则 / 手工指定）
- 识别方式（结构化提取 / OCR / 手工录入）
- 置信度
- 是否人工确认
- 是否人工修改（若是，记录修改前值、修改后值、修改时间）

#### Sheet 4：需求符合性矩阵（若有需求标准）

- 需求编号
- 需求分类
- 需求描述
- 是否必选
- 各供应商符合性状态（符合 / 部分符合 / 不符合 / 无法判断）
- 各供应商证据摘要（证据原文 + 来源位置）
- 是否人工确认

# 4. 用户流程

## 4.1 交互模型

项目工作台采用 **阶段化工作台** 模式，而非严格线性向导。用户可以在 5 个阶段之间自由切换，修改上游数据后，下游结果自动标记失效并提示重算。

### 5 个阶段

| 阶段 | 核心操作 |
|------|----------|
| 第一步：导入文件 | 上传文件、确认供应商名称、选择参与比价的表格 |
| 第二步：标准化 | 应用规则、列名映射确认、手工修正字段值 |
| 第三步：商品归组 | 查看候选归组、确认/拆分/合并/标记不可比 |
| **第四步：符合性审查（可选）** | 录入/导入需求标准、查看符合性矩阵、确认匹配结果 |
| 第五步：比价与导出 | 查看比价结果（含有效最低价）、处理异常、导出 Excel |

> **可选说明**：第四步「符合性审查」为可选模块。未录入项目需求标准时，自动跳过此步骤，直接进入比价。需求标准可在任何阶段录入/修改，录入后触发符合性匹配。

### 阶段状态机制

每个阶段有独立状态：

| 状态 | 含义 |
|------|------|
| 未开始 | 上游阶段尚未完成 |
| 已完成 | 阶段产物已生成且经用户确认 |
| 已失效（dirty） | 上游数据发生变更，当前阶段产物需重新确认/重算 |
| 已跳过 | 该阶段不适用（如无需求标准时的符合性审查阶段） |

**失效传播规则**：当用户修改某阶段数据时，该阶段之后的所有阶段自动标记为「已失效」。

```
修改标准化数据      → 归组 dirty → 符合性 dirty → 比价 dirty
修改归组            → 符合性 dirty → 比价 dirty
修改需求标准        → 符合性 dirty → 比价 dirty
修改符合性匹配结果  → 比价 dirty
新增/删除供应商文件 → 标准化 dirty → 归组 dirty → 符合性 dirty → 比价 dirty
```

> **决策说明**：MVP 采用整体失效策略，不做精确的行级影响追踪。实现简单、逻辑清晰。

### 回退场景

MVP 必须支持以下回退场景：

1. **归组阶段 → 标准化阶段**：发现字段标准化有误，回到标准化修正后，归组、符合性和比价结果标记失效
2. **符合性阶段 → 归组阶段**：发现归组错误，回到归组调整后，符合性和比价结果标记失效
3. **比价阶段 → 符合性阶段**：发现符合性判断有误，回到符合性调整后，比价结果标记失效
4. **新增供应商文件**：新文件单独走导入/标准化，完成后归组、符合性和比价阶段标记失效

### 数据层支持

项目数据中需维护阶段状态字段：

- `import_status` / `import_completed_at`
- `normalize_status` / `normalize_completed_at`
- `grouping_status` / `grouping_completed_at`
- `compliance_status` / `compliance_completed_at`（新增）
- `comparison_status` / `comparison_completed_at`

当上游数据变更时，自动将下游状态设为 `dirty`。

## 4.2 主流程步骤

1. 新建项目
2. 上传供应商报价文件
3. 确认供应商名称
4. 查看并选择参与比价的表格
5. 应用全局规则进行字段标准化
6. 手工修正识别和标准化结果
7. 生成商品归组候选
8. 用户确认商品归组
9. （可选）录入/导入项目需求标准
10. （可选）查看符合性矩阵，确认匹配结果
11. 生成比价结果（若有需求标准，计算有效最低价）
12. 导出 Excel 审计底稿

# 5. 关键业务规则

## 5.1 供应商确认规则

- 每个导入文件必须绑定供应商名称
- 供应商名称必须经用户确认
- 未确认供应商名称的文件不得进入比价阶段

## 5.2 商品归组规则

- 系统只生成候选归组，不直接做最终归组裁决
- 最终归组必须由用户确认
- 未确认归组的记录不得自动进入最终比价结论

## 5.3 税价口径规则

#### 可自动识别的场景（有限枚举）

系统仅在以下场景自动识别税价口径：

- 列名明确包含「含税单价」「不含税单价」「未税单价」等关键词
- 税率字段有明确数值（如 13%、6%、0%）
- 列名中同时出现「含税」和「不含税」两列，系统分别识别

#### 需人工确认的场景

以下场景系统不自动判定，标记为「税价口径待确认」：

- 仅有「单价」列，无法判断含税/未税
- 备注或其他字段中提及税率（如"含 13% 税"），但非独立列
- 单价和总价的含税口径可能不一致
- 税率字段缺失或为空

#### 异常处理

- 当同一商品组不同供应商的税价口径不一致时：系统标记异常，不自动给出「最低价结论」
- 税价口径未知时：必须提示人工复核
- 所有税价异常在「待处理问题清单」中集中显示

## 5.4 单位规则

- 当同一商品组单位不一致时：
  - 系统标记异常
  - 不自动比较价格
- MVP 不做复杂单位换算

## 5.5 币种规则

- MVP 默认单项目单币种
- 当出现多币种时：
  - 系统标记异常
  - 不自动比较价格
- MVP 不做汇率换算

## 5.6 低置信字段规则

- OCR/解析得到的低置信字段必须高亮提示
- 低置信字段可参与后续处理，但必须保留"待人工复核"状态
- 如用户确认采信，应记录确认行为

## 5.8 异常处理规则总表

以下表格统一定义各类异常的界面行为、对比价结论的影响和导出处理：

| 异常类型 | 触发条件 | 界面行为 | 是否阻断比价结论 | 是否允许导出 | 导出时标记方式 |
|----------|----------|----------|-----------------|-------------|---------------|
| 税价口径不一致 | 同一商品组供应商含税/未税口径不同 | 高亮异常行 + 问题清单 | 阻断该商品组的最低价结论 | 允许导出 | 该商品组标记「税价口径不一致，最低价仅供参考」 |
| 税价口径未知 | 无法判定含税/未税 | 黄色提示 + 问题清单 | 阻断该商品组的最低价结论 | 允许导出 | 该商品组标记「税价口径未知」 |
| 单位不一致 | 同一商品组供应商单位不同 | 高亮异常行 + 问题清单 | 阻断该商品组的价格比较 | 允许导出 | 该商品组标记「单位不一致，不可比」 |
| 币种不一致 | 项目中出现多种币种 | 全局提示 + 问题清单 | 阻断涉及商品组的价格比较 | 允许导出 | 相关商品组标记「币种不一致，不可比」 |
| 低置信字段未确认 | 字段置信度低于阈值且用户未确认 | 字段高亮 + 问题清单 | 不阻断，但标记 | 允许导出 | 该字段标记「低置信，未经人工确认」 |
| 商品未确认归组 | 归组状态为待确认 | 归组列表标记 + 问题清单 | 未确认项不进入比价结论 | 允许导出（不含未确认项） | 导出时注明「N 项未归组，未纳入比价」 |
| 未映射字段 | 原始列名无法匹配标准字段 | 列名标记 + 问题清单 | 不阻断 | 允许导出 | 追溯表中标记「未映射」 |
| 规则冲突 | 同一列名命中多条规则 | 列名标记 + 问题清单 | 不阻断（需人工选择后继续） | 允许导出 | 追溯表记录最终选择的规则 |
| 必填字段缺失 | 商品名称/单位/数量/单价为空 | 字段高亮 + 问题清单 | 不阻断，但标记 | 允许导出 | 该行标记「关键字段缺失」 |

> **设计原则**：所有异常均不阻断整体导出流程，但必须在导出的 Excel 中明确标记。用户看到的比价结论必须反映真实的数据质量状态，不得静默忽略任何异常。

## 5.7 手工修改留痕规则

所有手工修改必须记录：

- 修改字段
- 修改前值
- 修改后值
- 修改时间

# 6. 匹配规则管理要求

## 6.1 规则存储

- 所有规则保存在本地
- 与项目数据分离
- 关闭软件后下次打开仍有效

## 6.2 默认模板

系统内置 1 套默认模板，MVP 可扩展至少量附加模板，例如：

- 通用采购模板
- IT 设备模板

## 6.3 项目级覆盖

- 全局规则用于默认处理
- 项目内允许临时覆盖
- 项目级覆盖不自动回写全局规则，除非用户明确选择"保存到全局规则"

## 6.4 规则管理边界说明

#### 规则导入冲突处理

当导入的 JSON 规则文件中存在与本地已有规则同名（相同原始关键词 + 相同规则类型）的条目时：

- 系统逐条提示冲突，用户可选择「覆盖本地」或「跳过」
- 支持「全部覆盖」和「全部跳过」的批量操作
- 导入完成后显示汇总：新增 N 条、覆盖 N 条、跳过 N 条

#### 规则变更与已完成项目的隔离

- **已完成项目**的标准化结果、归组结果、比价结果固化在项目数据库（project.db）中，不会因后续全局规则变更而自动重算
- **未完成项目**如重新执行标准化，将使用当前全局规则（除非项目内已固化对应覆盖规则）
- 删除全局规则不影响已完成项目中的历史命中记录

#### 轻量规则快照（追溯留痕）

为支撑「规则变更不影响已完成项目」的可验证性，项目数据库中的规则命中记录应保存以下快照信息：

- 命中的规则 ID
- 规则名称
- 命中时的规则匹配内容摘要（如：原始关键词 → 标准字段，匹配方式）

> **说明**：MVP 不做完整的规则版本管理系统，但通过快照留痕确保历史项目可追溯「当时是按哪条规则识别的」。完整规则版本化列入后续版本方向。

#### 最小规则测试能力

规则编辑界面提供轻量测试功能：

- 用户输入一个测试列名（如「报价含税」）
- 系统实时显示该列名会映射到哪个标准字段
- 若命中多条规则，显示冲突提示及优先级裁决结果

> **说明**：此功能旨在降低规则维护的试错成本，不要求实现完整的规则测试框架。

# 8. 数据存储要求

## 8.1 存储架构

采用 **SQLite + 文件系统** 的混合存储方案。

### 全局数据（JSON 文件）

- 规则模板（JSON）
- 规则配置（JSON）
- 标准字段定义（JSON）
- 最近项目列表（JSON）
- 应用配置（JSON）

> **决策说明**：全局数据量小，不需要 SQL 查询能力。规则的导入/导出本身就是 JSON 格式，存储也用 JSON 减少序列化转换层，且用户手动备份/迁移规则更直观。

### 项目数据（每项目独立 SQLite + 文件目录）

每个项目为一个独立目录，包含：

- `project.db`（SQLite）：项目信息、供应商信息、解析结果、标准化结果、商品归组结果、比价结果、阶段状态、操作日志
- `source_files/`：原始导入文件副本
- `exports/`：导出的 Excel 文件

> **决策说明**：每项目一个 SQLite 文件，项目天然自包含，方便备份、删除、归档，也为后续「导出项目包」功能打好基础。项目内数据关联复杂（文件→表格→行→字段→规则命中→修改记录→归组→比价），SQLite 的关系查询能力是必要的。

### 本地目录结构

```text
app_data/
  config.json                  # 应用配置、最近项目列表
  rules/
    default-template.json      # 内置默认模板
    user-rules.json            # 用户自定义规则
  projects/
    <project_id>/
      project.db               # 项目 SQLite 数据库
      source_files/             # 原始文件
      exports/                  # 导出文件
```

## 8.2 最近项目

- 保存最近项目列表（存储在全局 config.json 中）
- 支持快速重新打开
- MVP 不要求复杂检索或归档管理

## 9.2 跨平台

- Windows
- macOS
- Linux

## 9.3 安全性

- 商业数据不出本地
- 适合企业内部安装使用

## 9.4 性能目标

MVP 不对所有文件类型承诺统一耗时指标，按类型和场景定义预期：

#### 文件解析耗时

| 文件类型 | 耗时预期 | 说明 |
|----------|----------|------|
| Excel（.xlsx） | < 3 秒 | 含多工作表场景 |
| Word（.docx） | < 3 秒 | 含多表格场景 |
| 数字版 PDF（结构化提取） | < 5 秒 | 单文件，10 页以内 |
| 数字版 PDF（Fallback OCR） | 10-60 秒 | 视页数而定，需显示进度 |
| 图片 / 扫描版 PDF（实验性） | 15-120 秒 | 视页数和清晰度而定，需显示进度并允许取消 |

#### 项目规模建议值

| 指标 | MVP 建议上限 | 说明 |
|------|-------------|------|
| 单项目供应商数 | 10 家 | 超过时提示但不阻断 |
| 单项目文件总数 | 30 个 | 超过时提示但不阻断 |
| 单文件表格数 | 20 个 | 超过时提示但不阻断 |
| 单次比价商品行数 | 500 行 | 超过时可能影响归组和导出性能 |

> **说明**：以上为建议值而非硬性限制。系统在接近上限时应给出提示，但不阻断用户操作。

## 9.5 精度目标

MVP 不承诺所有输入类型的全自动高精度结果，优先保证：

- 结构化结果可预览
- 可人工修正
- 可追溯
- 可导出

## 9.6 商业使用约束

- 所有引入组件必须支持商业使用
- 许可证需纳入依赖清单审查
- 不强制要求全部为 Apache 2.0

# 10. MVP 验收标准

## 10.1 基础流程验收

用户能够完成以下闭环：

- 新建项目
- 导入 3 家供应商文件
- 确认供应商名称
- 选择需要参与比价的表格
- 应用规则标准化字段
- 手工修正问题数据
- 确认商品归组
- 生成比价结果
- 导出 Excel 审计底稿

## 10.2 规则管理验收

- 规则可新增、编辑、删除
- 规则可永久保存
- 规则可导入、导出
- 项目中可使用全局规则
- 项目中新增映射可选择保存为全局规则

## 10.3 追溯能力验收

导出的 Excel 中可查看：

- 来源文件
- 供应商
- 表格来源
- 原始字段与标准字段关系
- 命中规则
- 手工修改记录
- 低置信标记

## 10.4 异常控制验收

当出现以下情况时，系统必须提示而不是静默处理：

- 税价口径不一致
- 单位不一致
- 币种不一致
- 商品未确认归组
- 低置信字段未确认

## 10.5 质量指标验收

#### 追溯完整率

- 导出 Excel 中每一行比价结果必须能追溯到来源文件、来源定位（sheet/页码/行列）和原始值
- 目标：**100%**（追溯链断裂视为缺陷）

#### 标准验收数据集

MVP 验收前应准备 **3-5 套标准测试数据集**，覆盖以下典型场景：

| 数据集 | 场景描述 | 验证重点 |
|--------|----------|----------|
| 数据集 1 | 3 家供应商，均为 .xlsx，格式规范，字段名一致 | 基础流程闭环、自动映射成功率 |
| 数据集 2 | 3 家供应商，混合 .xlsx + .docx + 数字版 PDF，字段名不一致 | 多格式解析、规则映射、手工修正 |
| 数据集 3 | 4 家供应商，含税价口径不一致、单位不一致 | 异常检测、异常标记、导出标记 |
| 数据集 4 | 3 家供应商，商品名称存在品牌别名和噪音词差异 | 归组归一化、置信度分层、禁止自动归组场景 |
| 数据集 5（可选） | 含 1 个扫描版 PDF 或图片 | 实验性 OCR 路径验证 |

每套数据集应预先定义 **预期结果**（预期映射结果、预期归组、预期异常标记），验收时与系统实际输出对比。

#### 方向性质量指标（基于标准数据集）

| 指标 | 含义 | MVP 方向性目标 |
|------|------|---------------|
| 字段自动映射命中率 | 系统自动映射正确的字段占比（数据集 1-2） | ≥ 80% |
| 高置信归组无需修改率 | 自动归为高置信的组中，用户确认时无需拆分/修改的占比 | ≥ 90% |
| 异常检出率 | 预设异常场景被系统正确标记的占比（数据集 3） | 100% |
| 追溯完整率 | 导出比价结果中可完整追溯到来源的占比 | 100% |

> **说明**：以上百分比目标基于标准验收数据集度量，不作为面向所有用户输入的通用承诺。实际用户场景中的效果取决于报价文件的规范程度和规则覆盖度。
